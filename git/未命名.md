## 一、仓库初始化与基础配置
### 1. 远程仓库创建
在 Git 平台（GitHub/GitLab/Gitee 等）创建项目仓库，完成以下初始化：
- **添加基础文件**：
    - `README.md`：项目说明文档（功能、技术栈、快速开始）。
    - `.gitignore`：根据项目语言/框架配置忽略规则（如 Node.js 的 `node_modules`、Python 的 `__pycache__`）。
    - `LICENSE`：开源协议（如 MIT、Apache），内部项目可省略。
- **设置默认分支**：
    - 若采用 Git Flow，设为 `develop`；若采用 GitHub Flow，设为 `master`/`main`。
### 2. 本地仓库初始化
```Bash
# 克隆远程仓库
git clone <repository-url>
cd <project-name>

# （可选）配置用户信息（企业内部需统一）
git config user.name "Your Name"
git config user.email "your.email@company.com"
```
### 3. 分支保护规则（关键！）
在 Git 平台设置**强制分支保护**，避免误操作：
- **主分支（****`master`****/****`main`****）**：
    - 禁止直接推送，必须通过 PR/MR 合并。
    - 至少 1-2 人审查通过。
    - CI/CD 流水线必须通过（单元测试、代码扫描等）。
- **开发分支（****`develop`****）**：
    - 同样禁止直接推送，需 PR 合并。
## 二、分支策略：Git Flow（企业标准方案）
Git Flow 是最经典的企业级分支模型，通过明确的分支职责实现版本管理，适合有固定发布周期的项目。
### 核心分支类型

| 分支类型      | 命名规范            | 从何而来      | 合并到何处                | 用途说明                                                       |
| --------- | --------------- | --------- | -------------------- | ---------------------------------------------------------- |
| **主分支**   | `master`/`main` | 初始化       | -                    | 存放**稳定可发布**的代码，每个提交对应一个正式版本（需打 Tag，如 `v1.0.0`）。            |
| **开发分支**  | `develop`       | `master`  | `master`             | 集成最新开发功能的分支，是**功能开发的基础分支**。                                |
| **功能分支**  | `feature/*`     | `develop` | `develop`            | 开发新功能/特性，如 `feature/user-login`、`feature/payment-gateway`。 |
| **发布分支**  | `release/*`     | `develop` | `develop` + `master` | 准备正式发布，用于**最后阶段的 Bug 修复、版本号调整**。                           |
| **热修复分支** | `hotfix/*`      | `master`  | `develop` + `master` | 紧急修复线上 Bug，如 `hotfix/fix-login-crash`。                     |

### Git Flow 操作流程
#### 1. 开发新功能（Feature）
```Bash
# 1. 从 develop 创建功能分支
git checkout develop
git pull origin develop  # 先同步最新代码
git checkout -b feature/user-login

# 2. 开发并频繁提交（遵循提交规范，见下文）
git add .
git commit -m "feat(auth): add login form UI"
git commit -m "feat(auth): implement login API integration"

# 3. 功能完成后，推送远程并创建 PR
git push origin feature/user-login
# 在 Git 平台创建 PR，目标分支为 develop，等待审查
```
#### 2. 准备发布（Release）
```Bash
# 1. 从 develop 创建发布分支（假设版本号 v1.0.0）
git checkout develop
git pull origin develop
git checkout -b release/v1.0.0

# 2. 修复 Bug、调整版本号（如修改 package.json、pom.xml）
git commit -m "chore: bump version to 1.0.0"
git commit -m "fix(ui): resolve mobile layout issue"

# 3. 完成发布：合并到 master + 打 Tag + 合并回 develop
git checkout master
git merge --no-ff release/v1.0.0  # --no-ff 保留分支历史
git tag -a v1.0.0 -m "Release version 1.0.0"

git checkout develop
git merge --no-ff release/v1.0.0  # 将发布阶段的修改同步回 develop

# 4. 推送所有变更和 Tag
git push origin master develop --tags
git branch -d release/v1.0.0  # 删除本地发布分支
```
#### 3. 紧急修复（Hotfix）
```Bash
# 1. 从 master 创建热修复分支
git checkout master
git pull origin master
git checkout -b hotfix/fix-login-crash

# 2. 修复 Bug 并提交
git commit -m "fix(auth): resolve crash on invalid token"

# 3. 完成修复：合并到 master + 打 Tag + 合并回 develop
git checkout master
git merge --no-ff hotfix/fix-login-crash
git tag -a v1.0.1 -m "Hotfix version 1.0.1"

git checkout develop
git merge --no-ff hotfix/fix-login-crash

# 4. 推送并删除分支
git push origin master develop --tags
git branch -d hotfix/fix-login-crash
```
## 三、简化方案：GitHub Flow（适合持续部署）
若项目为**持续部署**（如 Web 应用、SaaS），可采用更简洁的 GitHub Flow：
1. 只有一个长期分支 `master`/`main`，始终保持可发布状态。
2. 开发新功能时，从 `master` 切出 `feature/*` 分支。
3. 功能完成后提交 PR，审查通过后合并到 `master`，立即触发 CI/CD 部署。
4. 线上问题直接从 `master` 切 `hotfix/*` 修复，合并后立即部署。
## 四、提交规范：Conventional Commits
企业要求提交信息**结构化、可追溯**，需遵循「Conventional Commits」规范：
```Plain
<type>(<scope>): <subject>

<body>  # 可选，详细说明变更
<footer>  # 可选，关联 Issue、Breaking Change 等
```
### 关键参数说明
- **type**（必须）：提交类型
    - `feat`：新功能
    - `fix`：修复 Bug
    - `docs`：文档变更
    - `style`：代码格式调整（不影响逻辑）
    - `refactor`：重构（既不新增功能也不修复 Bug）
    - `perf`：性能优化
    - `test`：测试相关
    - `chore`：构建/工具链相关
- **scope**（可选）：影响范围，如 `auth`、`payment`、`ui`
- **subject**（必须）：简短描述（不超过 50 字符），用动词开头
### 示例
```Plain
feat(auth): add OAuth2 login support

- Implement Google/Facebook OAuth flow
- Add token refresh mechanism
- Update user model to store OAuth provider

Closes #456
```
## 五、企业协作最佳实践
1. **小步提交**：每个提交只做一件事，方便回滚和审查。
2. **频繁同步**：每天至少拉取一次 `develop`/`master`，减少合并冲突。
3. **PR 描述清晰**：说明“做了什么”“为什么这么做”“怎么测试”，降低审查成本。
4. **代码审查前置**：审查人需检查代码质量、逻辑、测试覆盖，提出修改意见。
5. **定期清理分支**：合并后立即删除已合并的分支，保持仓库整洁。
6. **Tag 管理**：每次发布到 `master` 都打**语义化 Tag**（`v<major>.<minor>.<patch>`，如 `v1.2.3`）。
